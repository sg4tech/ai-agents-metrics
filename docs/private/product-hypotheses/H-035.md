# History-derived retry pressure may be more reliable than manually-captured attempt counts

Single-hypothesis file.

### H-035 â€” History-derived retry pressure may be more reliable than manually-captured attempt counts

- Status: `active`
- Created: `2026-04-10`
- Linear: `CODEX-59`
- Statement:
  - The retry pressure signal currently stored in the live ledger (`events.ndjson`) may be systematically underreported because agents rarely log explicit retry attempts during normal work. History-derived retry counts from conversation session files may be a more reliable source of this signal than manual ledger entries.

## Evidence

A `history-compare` run on 2026-04-10 against the current project revealed:

| Signal | Ledger | History warehouse |
|--------|--------|-------------------|
| Closed goals / threads | 14 | 72 |
| Total attempts | 14 (1:1 ratio) | 118 |
| Threads with retry pressure | **0 (0%)** | **21 (29%)** |
| Total tokens | 747M | 2,085M |

The ledger shows perfect zero-retry history. The actual session history shows ~1 in 3 threads had retry pressure. This gap is not a data quality issue â€” it reflects the fundamental problem that agents almost never write explicit retry entries to the ledger; they simply continue working in the same session or open a new one.

## Why It Matters

Retry pressure is the secondary north-star metric (H-002). If it reads as zero when it is actually ~29%, the product is blind to its most important efficiency signal. Workflow changes that increase or decrease retry pressure would be invisible.

## Expected Upside

- Retry pressure in `show` reflects what actually happened, not just what was manually recorded
- Enables before/after comparisons of retry pressure across workflow changes
- Reduces the manual logging burden on agents â€” the ledger only needs boundaries and judgements (per H-008 minimal live contract), not attempt-by-attempt tracking
- Unlocks the full value of the hybrid H-008 approach for this specific metric

## Main Risks

- Session boundary â‰  ledger goal boundary: one thread in history may span multiple ledger goals, or one ledger goal may span multiple threads; naive merging will produce noise
- Worktrees: Claude Code creates separate project directories per worktree; these must not be double-counted as independent goals
- Retry definition gap: a "retry" in session history (a new session file for the same thread) is not identical to a "retry" in the ledger (a new attempt entry with an explicit failure reason); conflating them may overcount
- History warehouse may not be present or up-to-date; `show` must degrade gracefully to ledger-only behavior

## Alternatives Considered

- Stricter agent prompting to log attempts explicitly â†’ increases friction, proven not to work in practice
- Show both ledger and history retry counts side-by-side â†’ adds complexity without resolving the gap
- Replace ledger attempt tracking entirely with history â†’ loses the explicit failure-reason signal, which is still live-capture-only per H-008

## Current Confidence

- `medium-high` that history retry counts are more accurate than ledger counts
- `medium` that the sessionâ†’goal boundary alignment is solvable without false positives

## Evidence Status

- Supported by direct measurement: 0 ledger retries vs 21/72 history retry threads on the same project
- Not yet validated: whether history retry threads map cleanly onto ledger goals (one-to-one or many-to-one)

## Next Re-evaluation Trigger

- After a first integration attempt shows whether sessionâ†’goal alignment produces clean or noisy retry counts
- After checking whether worktree sessions are correctly excluded or merged

## Exit Criteria

- `show` reports history-derived retry pressure when the warehouse is available
- The retry count from history and the ledger are reconciled or the gap is explicitly explained in the output
- Worktrees are handled without double-counting
- A before/after comparison on this project shows whether the integrated signal is actionable

### Steps

**Step 1 â€” Worktree normalization in derive layer** (`history_derive.py`)

- Add `normalize_project_cwd(cwd: str) -> str` â€” strips `/.claude/worktrees/<name>` suffix
- Add `parent_project_cwd` column to `derived_projects`
- Aggregate retry stats by `parent_project_cwd` instead of raw `project_cwd`
- Update `history_compare.py` to filter by `parent_project_cwd`

Tests: `normalize_project_cwd` for plain path / worktree path; aggregation merges worktree threads into parent.

**Step 2 â€” History reader** (`history_compare.py`)

New function: `read_history_signals(warehouse_path, project_cwd) -> HistorySignals | None`

```python
@dataclass
class HistorySignals:
    project_threads: int
    retry_threads: int
    retry_rate: float
    ledger_goal_alignments: int
    ledger_goals_total: int
```

Logic:
- Warehouse absent â†’ `None`
- Project-level: read from `derived_projects` by `parent_project_cwd`
- Per-goal count: for each ledger goal with `started_at`/`finished_at`, count matching history threads in window

Tests: absent warehouse; project-level rate; per-goal window filtering; worktree merge.

**Step 3 â€” Wire into `show`** (`commands.py`, `cli.py`)

- Add `--warehouse-path` to `show_parser` (default: `RAW_WAREHOUSE_PATH`)
- In `handle_show`: call `read_history_signals(warehouse_path, cwd)` after `load_metrics`
- Pass `HistorySignals | None` into `print_summary` and `render_summary_json`
