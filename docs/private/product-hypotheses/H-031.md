### H-031 — GrowthBook may help measure which features are actually useful or useless by making experiments and feature adoption easier to observe

- Status: `active`
- Priority: `low`
- Created: `2026-04-06`
- Statement:
  - GrowthBook may give `codex-metrics` a practical experimentation and feature-flag layer that makes it easier to see which features improve real product outcomes and which features do not.
  - If that is true, the project could use GrowthBook to separate feature ideas that are genuinely helpful from features that only feel helpful in isolation.
- Why it matters:
  - The product still needs a better way to distinguish useful work from attractive but low-value work.
  - If feature-level experiments are easy to run, then product decisions can rely more on observed behavior and less on intuition alone.
  - That could reduce the risk of shipping or keeping features that do not improve agent workflow, analysis quality, or cost control.
- Expected upside:
  - make it easier to run feature experiments or staged rollouts
  - improve visibility into feature adoption, retention, and outcome impact
  - create a cleaner path for deciding whether a feature should stay, be revised, or be removed
  - help the team identify low-value features sooner
  - support more disciplined product iteration without relying only on anecdotal feedback
- Expected resource cost:
  - low-to-medium for evaluation and initial integration
  - medium if the project needs event instrumentation, experiment definitions, and analysis conventions to make the signals useful
- Main risks or where this may be wrong:
  - the product may be too small for a full experimentation platform to pay off yet
  - feature-flag tooling can add operational and conceptual complexity
  - GrowthBook may help with rollout control more than with proving true feature usefulness
  - if the core product signals are still weak, experiments may only measure noisy proxies
  - this likely stays lower priority than improving the core metrics, history, and analysis loop first
- Alternatives considered:
  - keep feature evaluation manual and rely on direct observation, retros, and issue-level feedback
  - add a smaller in-house flag or toggle layer instead of adopting a full experimentation platform
  - instrument only the specific product events needed for a few high-priority decisions
- Current confidence:
  - `low-to-medium`
- Current priority note:
  - low priority; useful to keep in view, but unlikely to be pursued in the near term
- Evidence status:
  - the product currently lacks a dedicated feature-experiment layer
  - the need is plausible because product value here is still being inferred from workflow outcomes, not from a mature feature analytics system
  - not yet validated by actual GrowthBook usage in this repository
- Evaluation plan:
  - identify one or two feature decisions where experiment-style evidence would change the outcome
  - test whether GrowthBook can measure those decisions with acceptable setup cost
  - compare the signal quality against simpler manual or issue-level evaluation
  - decide whether the tool is worth adopting before building deeper product instrumentation around it
- Next re-evaluation trigger:
  - after a concrete feature-valuation use case is defined
  - or after a lightweight experiment prototype shows whether the signal is actually actionable
  - or when the core product signals are stable enough that experimentation infrastructure becomes the bottleneck
- Related hypotheses:
  - relates to H-001 because useful features should be judged by outcome fit, not just delivery completion
  - relates to H-016 because any experimentation layer still depends on the right structured event logging underneath
  - may benefit from H-018 if conversation-text analysis is used to interpret experiment results
