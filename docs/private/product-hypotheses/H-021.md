### H-021 — A system-level immutable flag on the metrics file may reduce accidental edits, but the end-to-end privileged updater path is still unverified on both target OSes

- Status: `active`
- Created: `2026-04-03`
- Statement:
  - `metrics/codex_metrics.json` may be safer if the file is marked immutable at the operating-system level, but the update flow then needs a deliberately restricted privileged path because a normal script cannot both respect the lock and update the file without temporarily removing that protection.
  - On macOS, the closest primitive is `chflags uchg` for a user immutable flag, with `schg` as a stronger system immutable variant; on Linux, the closest equivalent is `chattr +i`.
  - What is already confirmed in this repository: the storage layer now uses a real OS-level immutability guard, and the tests cover the roundtrip path, the guard contract, and the CLI workflow.
  - What is still not confirmed: a production-ready constrained privileged updater path that has been exercised live on both macOS and Linux.
- Why it matters:
  - If true, the repository can reduce accidental manual edits and drift on the structured metrics source of truth.
  - The real product question is not “can we make it read-only?” but “can we make the updater safe, narrow, and auditable enough that the protection does not block the canonical write path.”
- Expected upside:
  - fewer accidental edits to the source-of-truth metrics file
  - clearer separation between read-only consumers and the one authorized writer
  - stronger protection against casual manual patching
  - a more obvious failure mode when something tries to mutate the metrics file outside the intended workflow
  - easier reasoning about drift because write access becomes intentional rather than ambient
- Expected resource cost:
  - low for the first pass if we only test the OS primitive and the updater path design
  - medium if we need a dedicated helper, sudoers rule, or launcher/daemon wrapper to keep the privileged write path narrow
  - medium if we also need cross-platform behavior across macOS and Linux
- Main risks or where this may be wrong:
  - there is no magic command that lets one script update a file while everyone else has no access unless the script itself has a separate privileged capability
  - overly strict immutability could make routine maintenance painful or brittle
  - the update flow may become more complicated than the protection is worth
  - file immutability protects against accidental mutation better than against a determined privileged user
- Alternatives considered:
  - keep using normal permissions and rely on the updater script plus tests
  - use a root-owned helper or restricted `sudoers` rule instead of file immutability
  - use a content-addressed or generated artifact flow so the writable source and the published read-only file are separate
  - keep the file writable but add a validation check that fails on unexpected manual edits
- Current confidence:
  - `medium-low`
- Evidence status:
  - supported by the existence of OS-level immutable flags such as `chflags uchg` on BSD/macOS-style systems and `chattr +i` on Linux
  - supported by the fact that `sudoers` can restrict execution to specific fully qualified commands, which is the practical way to create a narrow privileged updater path
  - not yet validated in this repository by measuring whether the privileged updater path actually works in a real macOS/Linux runtime without causing operational friction
- Evaluation plan:
  - keep the code-level immutable guard in place and verify it with tests
  - run a live macOS smoke test that proves the file can be unlocked, updated, and relocked with the intended primitive
  - run a live Linux smoke test that proves the same unlock-update-relock sequence with the Linux primitive
  - verify the narrow privileged updater path separately from the general storage API
  - compare the immutable-file approach with a root-owned helper and with plain permissions plus validation before promoting the hypothesis
