# Concurrent goal-id generation retrospective

## Situation

While working on multiple related tasks in the metrics repository, parallel execution exposed that new goal ids were being chosen outside the updater instead of being generated inside the source-of-truth script.

The visible symptom was repeated collisions on sequential ids such as `2026-03-29-066` and `2026-03-29-068` when more than one active flow tried to create a new goal around the same time.

## What happened

The repository workflow required every engineering task to be logged in `metrics/codex_metrics.json`.

In practice, new goal ids were manually supplied through `--task-id`. That worked in a single linear flow, but it broke down when multiple flows or agents were active at the same time:

- two flows could independently choose the same next id
- one flow could create a goal after another flow had already assumed that id was still free
- mutating updater commands had no write lock, so read-modify-write operations were not serialized

This created avoidable task bookkeeping friction and made the metrics tool less reliable exactly when operator concurrency increased.

## Root cause

The root cause was not just "manual ids are inconvenient".

The deeper issue was that goal identity allocation lived outside the authoritative updater, and mutating commands had no single-writer protection.

That meant the system had two coupled weaknesses:

- id allocation depended on operator guesswork instead of script-owned generation
- even internal generation would still be unsafe without locking because multiple processes could read the same pre-update state

## Retrospective

The bug was useful because it exposed a boundary mistake in the tool design.

The metrics updater is supposed to be the source of truth for task history, so it should also own identifier creation and mutation serialization. Requiring external callers to pick the next id was a leaky abstraction that only looked acceptable while usage stayed mostly sequential.

This is also a reminder that local CLI tools should still be designed for concurrency if the operating model includes multiple agents, retries, or overlapping implementation passes.

## Conclusions

- Goal ids should be generated by the updater when creating a new goal.
- Explicit `--task-id` should remain available for repair and migration flows, but it should not be the only creation path.
- Mutating commands need an exclusive lock around load, modify, validate, and save.
- Writes to metrics and reports should be atomic so readers do not observe partial files.

## Permanent changes

- Added internal goal id generation for new tasks in `scripts/update_codex_metrics.py`.
- Made `--task-id` optional for new goal creation while keeping explicit ids for existing update flows.
- Added an exclusive metrics mutation lock for `init`, `update`, `sync-codex-usage`, and `merge-tasks`.
- Switched metrics and report writes to atomic replace.
- Added regression tests for auto-generated ids, invalid no-id update usage, and parallel goal creation.
