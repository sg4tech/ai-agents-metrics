# `bgen` may need a modular architecture with separate layers for idea generation, scoring, validation, and evidence capture

Single-hypothesis file.

### H-002 — `bgen` may need a modular architecture that separates idea generation, automated scoring, validation planning, and real-world evidence capture

- Status: `active`
- Created: `2026-04-04`
- Statement:
  - `bgen` may work best if it is built as a modular pipeline rather than as one monolithic agent prompt, because idea generation, scoring, validation planning, and evidence capture are different jobs with different failure modes.
  - the system may need a canonical idea record that can move through states such as generated, scored, shortlisted, needs_validation, validating, validated, rejected, and promoted.
  - scoring may need to combine deterministic rules, structured features, and model-based judgment so the system can rank ideas automatically without turning the score into an opaque guess.
  - the validation layer may need to produce explicit next actions, such as the next market test, the minimum success metric, the data source, and the stopping rule.
  - the evidence layer may need to store user-reported outcomes and imported signals from external systems so the model can compare expectations with observed results over time.
- Why it matters:
  - If true, the product should be designed as a workflow engine first and a conversational interface second.
  - The architecture would make it possible to automate idea search and ranking while still leaving room for real-world testing and later learning.
  - The key product advantage would come from keeping the pipeline stateful, so the system remembers what was tried, what was predicted, and what actually happened.
- Expected upside:
  - automate repetitive idea triage instead of forcing manual review for every candidate
  - keep idea records normalized across generation, scoring, and validation steps
  - make it easy to compare ideas using the same schema and scoring rubric
  - support a feedback loop where real-world outcomes improve later scoring
  - allow future integrations such as Google Ads, analytics, CRMs, and landing-page tools to plug into a stable evidence model
- Expected resource cost:
  - medium-to-high, because the system needs a reliable data model, state transitions, scoring logic, and integrations rather than only a prompt or a single agent
  - medium ongoing maintenance cost if the scoring rubric, evidence schema, or integration contracts evolve frequently
  - potentially lower long-term cost than an ad hoc design, because the modular architecture should make the workflow easier to extend and debug
- Main risks or where this may be wrong:
  - the architecture may become too heavy before the product proves that people actually want the workflow
  - if the schema is too rigid, it may slow down exploration of new business idea formats
  - if scoring is split across too many mechanisms, the output may become hard to interpret
  - if integrations are built too early, the product may drift toward plumbing instead of decision support
  - a pure agentic chat interface may be quicker for first experiments, even if it is weaker as a long-term system
- Alternatives considered:
  - keep the first version as a single agent workflow and only later split into modules
  - build a scoring-only assistant and postpone validation and evidence capture
  - store ideas as simple notes instead of enforcing a full state machine
  - use external workflow tools for orchestration and keep `bgen` focused only on scoring
- Current confidence:
  - `medium`
- Evidence status:
  - supported by the need to automate search, scoring, and next-step suggestions without manual console work
  - supported by the need to capture real validation outcomes such as CAC checks and market-test results over time
  - not yet validated by implementation evidence, throughput benchmarks, or failure analysis of a simpler prototype
- Evaluation plan:
  - define the minimal idea schema that can survive generation, scoring, validation, and evidence capture
  - choose which parts of the pipeline must be deterministic and which can remain model-driven
  - test whether the architecture can rank ideas automatically with less manual intervention than a chat-first approach
  - compare a modular pipeline against a thin agent wrapper around shared storage
  - add integrations only after the core state model and scoring flow are proven useful
- Next re-evaluation trigger:
  - after the data model for an idea record is drafted
  - or after a first automated scoring pass is compared with human judgment
