# Higher agent autonomy may improve result quality and lower cost by separating discovery from delivery

Single-hypothesis file.

### H-019 — Higher agent autonomy may improve result quality and lower cost by separating discovery from delivery

- Status: `active`
- Created: `2026-04-02`
- Statement:
  - Giving the agent more permission to proceed independently, make reasonable assumptions, and avoid asking low-value clarification questions may improve the final result quality while also lowering cost, especially if the workflow is split into two modes: `discovery` for gathering and clarifying requirements, and `delivery` for immediately turning the confirmed requirements into ideas, solution shape, and execution.
- Why it matters:
  - If true, the workflow should optimize for decisive execution rather than for constant human confirmation, because unnecessary back-and-forth can create both delay and avoidable token cost.
  - A phase split would also let us optimize for different things at different times instead of forcing one metric set to serve both requirement discovery and implementation delivery.
- Expected upside:
  - fewer interruptions for questions that do not change the real outcome
  - lower token and time cost from reduced clarification churn
  - faster movement from task framing to implementation and verification
  - better end-to-end quality if the agent can keep momentum instead of stalling on minor unknowns
  - clearer phase-specific behavior so the agent knows when to gather requirements and when to execute them
  - a cleaner handoff from requirement gathering into immediate idea formation and implementation once the key constraints are clear
  - better metric alignment if `discovery` and `delivery` use different success criteria
- Expected resource cost:
  - low for the first pass, because the main change is behavioral framing and evaluation
  - medium if we later need stronger guardrails, prompt rules, or automatic detection of when a question is actually necessary
  - medium if we later formalize the two-mode boundary or need mode-specific reporting
- Main risks or where this may be wrong:
  - too much autonomy may cause the agent to make the wrong assumption and ship adjacent work
  - fewer questions could hide ambiguity instead of resolving it, especially on high-risk tasks
  - cost might go down while quality also drops, so the hypothesis must be checked against outcome fit rather than question volume alone
  - the right autonomy level may differ by task type, so a single global rule may be too blunt
  - the `discovery`/`delivery` split may add process overhead if we turn it into bureaucracy instead of a practical phase distinction
- Alternatives considered:
  - keep the current more-question-heavy interaction style and treat clarification as the safer default
  - allow autonomy only for low-risk tasks and keep stronger human confirmation for everything else
  - encode a narrow question budget instead of aiming for broad autonomous execution
  - keep a single unified workflow and accept that one metric set is always a compromise
  - use the mode split only as an internal mental model and not as an explicit tracked concept
- Current confidence:
  - `medium-low`
- Evidence status:
  - supported by the user's direct request to make the agent work as independently as possible
  - supported by the general intuition that avoidable clarification adds friction and cost
  - supported by the user's clarification that the workflow should likely have two modes with different optimization metrics
  - not yet validated by measured before/after data on result quality, retry pressure, or cost
- Evaluation plan:
  - compare goal outcomes where the agent proceeded with reasonable assumptions against outcomes where it asked more questions
  - measure whether fewer clarification questions correlate with better or worse exact-fit results, retry pressure, and cost per success
  - check whether the agent still escalates on truly decision-critical ambiguity while skipping low-value interruptions
  - check whether the agent can move from requirement gathering directly into solution ideas without an extra clarification loop when the missing detail is not decision-changing
  - look for task types where higher autonomy helps and task types where it backfires
  - compare discovery-mode tasks on requirement completeness and clarity, then compare delivery-mode tasks on execution quality and cost
  - check whether separating the modes reduces missed requirements in discovery and adjacent work in delivery
- Working metric set:
  - `discovery`: requirements completeness, ambiguity remaining, shared understanding
  - `delivery`: exact fit, retry pressure, cost per success
  - this set is provisional and should be revised if real usage shows better metrics or if any metric turns out to be too noisy, too expensive to measure, or not decision-useful
- Next re-evaluation trigger:
  - after we have enough real work history to compare autonomous execution against question-heavy execution
  - or when a concrete failure shows that the agent over-assumed instead of escalating
  - after we have real examples of both discovery and delivery in the same workflow
- Mode switch rule:
  - switch from `discovery` to `delivery` when requirements and success criteria are clear enough that the next useful step is execution or concrete solution shaping rather than clarification
  - if the remaining unknowns can change the requested outcome, stay in `discovery`
