# Event sourcing may eliminate git merge conflicts on the metrics file while preserving history and sync

Single-hypothesis file.

### H-030 — Event sourcing may eliminate git merge conflicts on the metrics file while preserving history and sync

- Status: `active`
- Created: `2026-04-06`
- Statement:
  - `metrics/codex_metrics.json` is currently a mutable JSON file tracked in git. Each `codex-metrics` CLI invocation mutates it in place. This creates two structural conflicts:
    1. The OS-level `uchg` immutability flag (set by the application guard) blocks `git pull` / `git merge` because git cannot unlink and replace a protected file.
    2. Even without the immutability flag, concurrent agents or users updating the file on separate branches will produce JSON merge conflicts on every pull.
  - The root cause is that a frequently-mutating derived state (a metrics database) is stored as a single mutable file in a system designed for text-diff merges.
  - Switching to an **append-only event log** as the canonical storage would eliminate both problems structurally:
    - Each CLI write appends a new event record (JSON line or small file) rather than mutating a shared document.
    - `codex_metrics.json` becomes a derived/cached artifact, regenerated from the log — it is no longer tracked in git.
    - Append-only entries from two branches never conflict: git sees only new lines added at the end of the log.
- Why it matters:
  - The current design makes multi-agent or multi-machine workflows unreliable. Any parallel work that ends in a `git pull` risks a blocked or conflicted merge.
  - The immutability guard (H-021) was intended to protect against accidental edits, but it makes the git workflow fragile as a side effect. Event sourcing would make the guard unnecessary on the tracked artifact.
  - History and auditability are preserved naturally: the event log is the history, not the derived state.
- Expected upside:
  - eliminate the class of git merge failures on `metrics/codex_metrics.json`
  - preserve full metrics history in a diff-friendly, append-only format
  - make multi-agent and multi-machine sync reliable without custom git tooling
  - simplify the immutability story: the derived file is not tracked, so OS-level flags on it do not affect git
- Expected resource cost:
  - high — requires reworking the storage layer, the CLI read/write path, and potentially the schema
  - existing `codex_metrics.json` history would need a one-time migration or a compatibility reader
- Main risks or where this may be wrong:
  - append-only log can still conflict if two agents append to the same file in the same git push window (rare, but possible with line-level conflicts at the exact same position)
  - regenerating derived state from a long event log may add latency to CLI commands
  - schema evolution becomes more complex: old events must remain readable under new schema versions
  - migration cost is non-trivial; existing tooling assumes the current single-file structure
- Alternatives considered:
  - **Remove uchg flag only** — eliminates the OS-level block but leaves JSON merge conflicts on every concurrent edit. Not a solution.
  - **Custom git merge driver** — teaches git to merge `codex_metrics.json` semantically (by task ID, by timestamp). Reduces conflicts but requires per-machine setup and breaks under schema changes. Workaround, not a structural fix.
  - **External database (SQLite, etc.)** — separates the store from git entirely. Solves sync differently (not via git), but loses git as the history and sync mechanism — incompatible with goal (2).
  - **Separate metrics git branch** — history in a dedicated ref, not in the working tree. Preserves git history and reduces main-branch conflicts, but adds tooling complexity and non-standard workflow.
- Current confidence:
  - `medium` — the direction is principled, but migration cost and append-conflict edge cases need validation before committing to implementation
- Evidence status:
  - `2026-04-06`: problem identified during a `git pull` failure caused by `uchg` flag on `metrics/codex_metrics.json`. Analysis confirmed the conflict is structural, not incidental.
- Next re-evaluation trigger:
