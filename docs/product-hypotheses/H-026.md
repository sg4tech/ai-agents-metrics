# Full-suite automated tests may lower development cost by shrinking the feedback loop and token spend

Single-hypothesis file.

### H-026 — Full-suite automated tests may lower development cost by shrinking the feedback loop and token spend

- Status: `active`
- Created: `2026-04-04`
- Statement:
  - If the full current automated test suite runs materially faster, then day-to-day development may get cheaper and easier because the feedback loop shortens, fewer minutes are spent waiting on verification, and the token cost per iteration may drop.
  - The current working target is a 50% runtime reduction for the full suite, measured against the current average test runtime.
  - token cost is treated as the first-priority proxy for development overhead, followed by wait time, then manual switching friction.
- Scope:
  - apply the hypothesis to this repository only
  - consider the entire suite, not just a narrow subset of tests
  - do not change the external CLI interface as part of the speed-up work
- Measurement:
  - use the current `codex-metrics` pipeline for the eventual before/after comparison
  - treat suite runtime as the main execution-time metric during the experiment
- Why it matters:
  - If true, test speed is not just a comfort improvement; it is a direct lever on iteration cost and team throughput.
  - The best optimization target may be the slowest repeated check if that check dominates the loop between a change and a trustworthy result.
- Expected upside:
  - shorter verify-fix cycles during active development
  - lower token spend per completed task if fewer repeated interactions are needed while waiting for or reasoning about test results
  - less context switching caused by slow feedback
  - better developer experience when the system can be checked more often without a large time penalty
- Expected resource cost:
  - medium, because speeding up tests may require fixture work, test splitting, caching, selective execution, or broader test-harness maintenance
  - possible tradeoff risk if speed improvements reduce coverage or make failures harder to trust
- Main risks or where this may be wrong:
  - token cost may not be the right proxy for development cost if the dominant savings come from wall-clock time instead
  - the slowest tests may not be the ones that matter most for the product loop
  - faster tests could still fail to improve outcomes if the bottleneck is unclear requirements, not verification latency
  - if speed work harms reliability, the apparent savings may be fake
- Alternatives considered:
  - keep test speed as a hygiene concern rather than a product-level hypothesis
  - optimize only the most expensive test tier instead of the whole suite
  - measure wall-clock iteration time and token spend separately instead of using one as a proxy for the other
  - focus on reducing test flakiness before reducing raw runtime
- Open questions:
  - which speed-up techniques are acceptable if they preserve quality
  - what minimum improvement threshold should count as success if 50% proves unrealistic
  - whether the most useful signal comes from raw runtime, token spend, or a combined score
- Current confidence:
  - `medium-low`
- Evidence status:
  - the hypothesis is plausible because slower feedback usually increases iteration overhead
  - no local measurement in this repo yet shows how much token cost is attributable to test latency specifically
  - the best signal may come from comparing token spend and elapsed iteration time before and after a targeted speed-up
- Evaluation plan:
  - identify the current slowest repeated automated checks
  - measure token cost and wall-clock time for the relevant development loop before any optimization
  - reduce the runtime of one high-frequency test path
  - compare whether the faster path reduces retries, context churn, and total token spend per completed change
  - keep coverage and failure fidelity constant while testing the speed change
- Next re-evaluation trigger:
  - after a measured before/after test-speed experiment exists
  - or after the repo shows repeated development sessions where test latency clearly dominates token usage

  - related to [H-019](/Users/viktor/PycharmProjects/codex-metrics/docs/product-hypotheses/H-019.md) because higher autonomy only helps if verification remains fast enough to keep the loop moving
